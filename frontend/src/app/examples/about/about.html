Sails.js + AngularJS Boilerplate

This is an example boilerplate for your Sails.js + AngularJS combo project. You can use this boilerplate as a starting point of your own awesome modern web application. In addition there are many example to show you how to get started.

Main goals of this boilerplate

  • Totally separated back- and frontend development
  • Really light frontend application
  • Provide an example application of sails.js + angular to community
  • Secure authentication with backend
  • Power of AngularJS + WebSockets
  • Simple examples of how to use these

Below is detailed information about this boilerplate.

"Sides" of this boilerplate

  • Backend
    • Only REST API, no frontend stuff - really just an API, nothing else.
    • Passport.js for user authentication. Supported auth types: local, twitter, facebook, google+, etc.
    • JWT support for API calls. JSON Web Tokens are used to to authenticate the user after a successfull login.
    • CORS support. Really that is what we need to do when separating back- and frontend.
    • Real business logic and validations for data Backend handles data related validations, logging, and user authentication.
  • Frontend
    • A single-page application (also known as single-page interface or SPI) is a web application or web site that fits on a single web page with the goal of providing a more fluid user experience akin to a desktop application.
    • Lightweight only client side application
    • Authentication to backend
    • Route examples for views
    • Support for WebSocket communication with backend
    • Interceptor for HTTP / Socket communication

Why do all this?

  • This is "the new way" to build modern web applications. Although there are different opinions about this - of course.
  • Main reasons
    • I'm interested in the architecture and structure of modern web applications - and this is my opinion of it.
    • To help the community to do stuff like this. These kind of full examples dont exist.
  • Pros
    • One language (javascript) to rule both sides. Easier to code, you can use the same code on both sides (eg. validation rules).
    • Developers can focus on the "side" that is familiar. This will make it easier for new people to work with your project.
    • Totally separated back- and frontend sides. These are totally separate applications that communicate with each other via the specified API.
    • Easier deployments. You can deployment back- and frontend applications at different times.
    • Ability to build multiple frontend applications for your backend. You can use different frontend frameworks: angular, ember, knockout, etc.
    • Ability to change your backend. If you get some real-show-stopper bugs / features / limitations with sails.js backend, you can change that to anything you like: C#, PHP, Ruby, etc. Just implement your API the same way - nothing else.
  • Cons
    • Complex structure of application.
    • More expensive to maintain that traditional web application. Basically at minimum two servers to run this.
    • API is the thing - so be very carefully when making changes. It is easy to implement versioning to your API endpoints though.
